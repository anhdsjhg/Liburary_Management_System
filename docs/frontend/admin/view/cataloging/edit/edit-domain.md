```md
# Cataloging/Edit/Edit.vue — Technical Documentation

## Overview
`Edit.vue` is the core MARC21 cataloging editor. It is responsible for loading, structuring, editing, validating, and saving bibliographic metadata. It is the most complex component in the system and acts as a domain controller for the entire cataloging workflow.

---

## Responsibilities
- Load MARC material data (GET material endpoint)
- Transform flat MARC data into structured sections/tags
- Provide inline editing of MARC fields
- Handle authority-controlled dropdowns
- Support Google Books import
- Export XML and print call numbers
- Manage full edit lifecycle (edit → validate → save → complete)
- Coordinate Preview, Import, and Log modals
- Maintain keyboard navigation inside MARC table

---

## Architecture Type
⚠️ GOD Component (Domain Controller Anti-pattern)

Combines:
- UI rendering
- business rules (MARC validation)
- data transformation (sectioning logic)
- API orchestration (10 endpoints)
- DOM manipulation (keyboard handling)
- Vuex mutation (direct state mutation)
- external API calls (Google Books)

---

## Data Model Flow

```

API (flat MARC records)
↓
edit_info[]
↓
divideIntoSections()
↓
divideIntoSubsections()
↓
sectioned[]
↓
UI (tagSelected.data)
↓
v-model mutation (direct API object mutation)
↓
joinSections()
↓
POST save

````

---

## API Dependencies

| Endpoint | Purpose |
|----------|--------|
| GET /cataloging/material/{type}/{id} | Load MARC data |
| GET /cataloging/authority | Authority dropdowns |
| GET /cataloging/material/genres | Genre data |
| GET /publisher/names | Publisher list |
| POST /cataloging/material/{type}/{id}/edit | Save record |
| GET /cataloging/material/export/{type}/{id} | XML export |
| GET /cataloging/material/print/{type}/{id} | Print PDF |
| POST /cataloging/material/{type}/{id}/complete | Complete status |
| DELETE /arrivals/delete-image/... | Image removal |
| GET Google Books API | External import |

---

## Vuex Coupling (CRITICAL)

Direct mutations:
```js
this.cataloging.image = ...
this.cataloging.id = ...
this.cataloging.type = ...
````

Issues:

* Bypasses Vuex mutations
* Invisible state tracking
* Breaks devtools traceability

---

## Major Business Rules

### MARC validation

* `245.a` (title) is required
* If `010.a` exists → `010.c`, `010.n`, `010.s`, `260.c` required

### Multi-value fields

* MARC repeatable fields handled via `repeatable` flag:

  * `1` → repeatable
  * `undefined` → duplicate entry
  * `null` → cleared state

---

## Critical Bugs

### 1. Memory leak (SEVERE)

```js
addEventListener('keydown', ...)
```

* Added repeatedly
* Never removed
* Grows per interaction

---

### 2. Sectioning logic bug (SEVERE)

```js
sections[char].info.push(info)
```

* Uses string index on array
* Misplaces MARC data
* Breaks numeric section mapping

---

### 3. Reactive mutation risk

* v-model directly mutates API objects
* No draft state
* No rollback mechanism

---

### 4. Navigation guard bug

* `this.saved` is single-use
* Does not reset after edits

---

### 5. External API bypass

* Uses `fetch()` instead of axios
* Bypasses interceptors
* No centralized error handling

---

## Performance Issues

* Recomputes sections on every load
* Rebuilds full MARC tree synchronously
* DOM listeners accumulate exponentially

---

## Side Effects

* DOM event listener injection
* Vuex mutation bypass
* Global loading state toggling (5 sources)
* Document class manipulation (overflow hidden)
* External API calls (Google Books)

---

## Migration Strategy

### Phase 1 (Critical Fixes)

* Fix `addKeyDownListener()` cleanup
* Fix `divideIntoSections()` indexing bug
* Reset `this.saved` on edit

### Phase 2 (State safety)

* Introduce draft model (`structuredClone`)
* Remove direct Vuex mutations
* Replace v-model on API objects

### Phase 3 (Architecture split)

Split into:

* MarcEditor.vue
* CatalogingToolbar.vue
* CatalogingFooter.vue

### Phase 4 (Composable extraction)

* useMarcEditor()
* useGoogleBooksApi()
* useKeyboardNavigation()

---

## Risk Level

🔴 CRITICAL (highest in system)

````

# Cataloging/Edit/Preview.vue — Technical Documentation

## Overview
Preview.vue renders a full bibliographic preview of MARC data in catalog-card format. It acts as a modal-driven read-only representation of structured MARC data generated by Edit.vue.

---

## Architecture Type
Hybrid Component:
- Composition API (`setup`)
- Options API (`data`, `methods`, `mixins`)
⚠️ Mixed reactive systems (legacy + modern)

---

## Data Flow

````

props.info (sectioned MARC data)
↓ watch()
mapFieldsToBookData()
↓
bookData (ref)
↓
UI rendering (catalog card view)

````

---

## Vuex Dependencies

Read-only:
- cataloging.image
- cataloging.id
- cataloging.type
- cataloging.type_key (⚠ potentially undefined)

No mutations (safe but coupled)

---

## API Usage

| Endpoint | Purpose |
|----------|--------|
| DELETE /arrivals/delete-image/{type}/{id} | Remove cover image |
| POST /arrivals/change-image | Upload image |

---

## Major Design Features

### Field Mapping System
- Converts MARC sections → display fields
- Includes:
  - authors
  - publishers
  - subjects
  - chapters
  - physical metadata

---

## Critical Issues

### 1. Mixed Vue API (HIGH RISK)
- setup() + Options API coexist
- Two reactivity systems in one component

---

### 2. Stale Vuex dependency
```js

````

* snapshot value
* becomes stale after updates

---

### 3. Missing type_key safety

* used in deleteImage()
* may be undefined → API failure risk

---

### 4. Memory leak risk

* document class manipulation not centralized

---

### 5. Dead code

* unused imports: Print, Save

---

## Performance Notes

* Multiple `.find()` over same dataset
* Recomputes mapping on every watch trigger

---

## Side Effects

* DOM overflow lock

* Modal lifecycle coupling

## Migration Strategy

### Phase 1

* Remove Options API completely

### Phase 2

* Convert to:

### Phase 3

* Replace mixins with composables

---

## Risk Level

🟠 HIGH (architecture inconsistency)

````
# ImportFromGoogleAPI.vue — Technical Documentation

## Overview
Handles selection and merging of MARC fields imported from Google Books API into existing catalog records.

---

## Architecture Type
Pure transformation UI component

- No API calls
- No Vuex coupling
- Data mutation-driven

---

## Data Flow

````

props.data (Google API mapped fields)
↓ sort()
UI selection
↓
importIt(selected)
↓
merge into edit_info[]
↓
callback(after)

````

---

## Key Behavior

### Import Logic
- If field has existing data:
  → duplicate entry via copy()
- If empty:
  → direct assignment

---

## Hidden Contracts

### 1. is_added flag
Used by Edit.vue to clean imported entries before save

### 2. repeatable flag logic
- controls MARC duplication behavior
- shared implicitly across components

---

## Critical Issues

### 1. Prop mutation (HIGH RISK)
```js
props.data.sort()
````

---

### 2. Missing null safety

```js
edit_info.find(...)
```

* can return undefined → crash

---

### 3. Fake TableDiv props

```js
:link="'link'"
:commit="'commit'"
```

* placeholder contract hack
* disables internal TableDiv actions

---

## Side Effects

* Direct mutation of parent edit_info
* Global copy() dependency
* Implicit MARC state modification

---

## Migration Strategy

* Replace prop mutation → computed sorted copy
* Replace copy() → structuredClone
* Replace callback → emit event
* Formalize is_added constant

---

## Risk Level

🟡 MEDIUM (data integrity risk)

